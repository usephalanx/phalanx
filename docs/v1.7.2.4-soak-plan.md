# v1.7.2.4 — 24-48h soak testing plan

**Goal**: prove v1.7.2.4 is stable enough to point at customer-facing repos for the marketplace launch. The DoD run (humanize Path 1) was a single proof point. Soak generates the *distribution* of behavior under varied failure modes.

**Soak duration**: 24h minimum, 48h target. Extend if any soft-signal threshold is breached.

**Hosts**: prod stack at `ubuntu@44.233.157.41`, testbed `usephalanx/phalanx-ci-fixer-testbed`, humanize fork `usephalanx/humanize`.

---

## 1. Success criteria (marketplace go/no-go)

These are the **hard thresholds**. Missing any one = soak fails, no marketplace launch until addressed.

| Threshold | Why |
|---|---|
| **Zero false ships** — every SHIP run has all GitHub checks green on the head SHA | The whole reason v1.7.2.4 exists |
| **Zero sha mismatches on SHIP** — `engineer_commit_sha == verified_commit_sha` for every SHIPPED row | The v1.7.2.3 invariant; if it ever breaks the gate's foundation is gone |
| **Zero unsafe ships** — no shipped commit contains a skip directive, deletes a test file, or edits CI config | The patch-safety contract |
| **≥ 95% of runs reach terminal state without operator intervention** | Self-healing/self-failing — no zombies, no manual cleanup needed |
| **≥ 80% SHIP rate on testbed cells** | Internal regression baseline |
| **≥ 50% SHIP rate on humanize re-derivations** | External proof point baseline (harder than testbed by design) |
| **0 runs hit `runtime_cap_exceeded`** | If this fires, something is hanging that we don't understand |
| **0 commander exceptions in logs** (excluding `slack_notifier.no_channel_for_run` and the known fingerprint_persist UUID warning) | Fail-loud principle |
| **Mean iterations to ship ≤ 1.4 on testbed**, **≤ 2.0 on humanize** | Bounded fix attempts |
| **Mean cost per SHIP ≤ $4** | Roughly TL+Challenger+Engineer at typical lengths; runaway cost is its own bug |

**Soft signals** (review-at-end, not auto-fail):

- Patch safety blocks fire on > 30% of attempts
- TL confidence < 0.5 on > 20% of runs (suggests TL prompt drift)
- Gate `poll_seconds` mean > 60s (slow CI on real repos)
- Mean tokens/run > 50k

## 3. Run cadence

**Total runs target**: ~120-200 over 48h (depending on duration of each).

**Schedule** (looped):

```
Hour bucket  | Workload                                | Approx runs
─────────────┼─────────────────────────────────────────┼────────────
Every 15 min | Random testbed cell (S1-S9)             | ~4/hour
Every 6h     | humanize Path 1 (S10)                   | ~4-8 over 48h
Every 12h    | humanize Path 2 (S11, rotated commit)   | ~2-4 over 48h
```

**Cell rotation**: weighted random, not pure round-robin. Heavier weight on the new scenarios (S5-S9) for first 12h since they're untested. Then balanced.

```
First 12h:  S1=10%, S2=10%, S3=10%, S4=10%, S5=15%, S6=15%, S7=15%, S8=10%, S9=5%
After 12h:  S1-S4=15% each, S5-S9=8% each
```

**Sandbox cleanup**: every 30 min, prune `cifix-v3-*` containers older than 1h. Keeps the host from drowning in detached sandboxes.

## 4. Soak runner architecture

Build a single Python script `scripts/v3_soak_runner.py` (~150 LOC). Responsibilities:

```
loop:
    1. Choose next scenario (weighted random per schedule above)
    2. Pre-flight:
         - prune stale cifix-v3-* sandboxes via SSH
         - verify integration cifixer_version is still 'v3'
    3. Run the trigger script (v3_python_regression.sh <cell> | v3_path1_humanize_tz.sh)
    4. On terminal status:
         - Pull run row + tasks rows from prod DB
         - Pull GitHub check-runs for the engineer head SHA
         - Compute derived metrics (iter count, fingerprints, gate decisions)
         - Emit one JSONL line to soak-runs.jsonl
    5. Sleep until next 15-min boundary
    6. SIGTERM-safe: write PID + last-run state to soak-state.json
```

Kill switch: `touch /tmp/soak-stop` between iterations exits cleanly. Resume via re-running the script — picks up next slot.

Output files:
- `soak-runs.jsonl` — one JSON row per run; primary metric source
- `soak-state.json` — current schedule position + last run id (for resume)
- `soak-runner.log` — runner's own diagnostic log (separate from prod logs)

The runner does NOT install anything on prod; it only triggers existing scripts and reads via SSH. Same access pattern as today's manual triggers.

## 7. Alert conditions

### Pager-level (immediately stop the soak; investigate)

- ANY false ship: `final_status=SHIPPED` AND any GitHub check on `head_sha` is `failure` after a ≥ 60s settle period
- ANY sha mismatch on SHIP: `verified_commit_sha != engineer_commit_sha`
- ANY unsafe ship: shipped commit's diff contains a regex match for skip directives, test deletion, or path under `.github/workflows/`
- Any commander exception traceback in worker logs (excluding the known `fingerprint_persist_failed` UUID-format warning)
- 3 consecutive runs hit `runtime_cap_exceeded` (suggests deep hang)

### Soft-signal (log, don't abort; review at soak end)

- SHIP rate < 70% in any 6-hour window
- Gate `poll_seconds` p95 > 240s (matrix CI getting slow)
- Cost per SHIP > $5
- Patch safety violation rate > 30%
- TL confidence < 0.5 on > 20% of runs

## 8. Step-by-step plan

### Pre-flight (T-1h to T-0)

1. Build `scripts/v3_soak_runner.py` + `scripts/v3_soak_dashboard.py` (~3-4 hours one-time work)
2. Author 5 new patches: `05-multi-failure.patch`, `06-importerror.patch`, `07-syntax.patch`, `08-ci-config.patch`, `09-hung.patch` in `~/phalanx-ci-fixer-testbed/failures/`
3. Pick 3 candidate humanize commits for S11 rotation (e.g. `git log --oneline src/humanize/time.py | head -20`)
4. Verify integrations are on `cifixer_version=v3` for testbed + humanize
5. Set total cost budget on the runner (default $200/48h)
6. Open the dashboard in a tmux pane

### Hour 0-12 (heavy new-scenario weighting)

- S5-S9 weighted up; the goal is to find untested-path bugs early
- Manual review every 2 hours: tail the dashboard, scan last 10 runs for anomalies
- If a pager alert fires → SIGTERM the runner, drop into investigation
- If 12h passes clean → flatten the weights for the next 12-36h

### Hour 12-24 (steady state)

- Standard mix
- Once-per-shift (every 8h) manual sanity check: pick one SHIPPED run at random and read the diff on GitHub. Plausible? Functionally correct?
- Once-per-shift: check `runs.error_context` for any FAILED/ESCALATED rows; verify the escalation_record has the structure we built

### Hour 24-48 (extended soak — only if criteria still passing)

- If we want a longer soak run, continue
- Otherwise, declare and stop at 24h with the data collected

### Decision point at end

Compute the metrics. If every hard threshold met:
1. Tag this soak: `git tag -a v1.7.2.4-soak-<date> -m "<summary>"`
2. Commit the JSONL data + dashboard snapshot to `docs/soak-results/<date>/`
3. Write `docs/soak-decision-<date>.md` with the go/no-go and reasoning
4. If GO: flip Challenger to gating mode, then stage marketplace launch

If any hard threshold breached:
1. STOP — no marketplace until fix
2. File a bug per failure mode observed
3. Patch + redeploy + re-soak (24h minimum re-test)

## 9. Expected signal patterns (what "healthy" looks like)

For interpretability of the live dashboard, here's what we *expect* to see:

| Signal | Healthy | Suspicious | Critical |
|---|---|---|---|
| TRUE_GREEN share | 80-90% | 60-80% | < 60% |
| NOT_FIXED → REPLAN share | 5-15% | 15-25% | > 25% (TL is consistently misdiagnosing) |
| REGRESSION share | 0-2% | 2-5% | > 5% (engineer keeps breaking things) |
| Patch safety blocks | 5-15% | 15-30% | > 30% (TL's plans drifting unsafe) |
| Mean iters (testbed) | 1.0-1.4 | 1.4-1.8 | > 1.8 |
| Mean iters (humanize) | 1.0-2.0 | 2.0-2.5 | > 2.5 |
| Gate poll_seconds (testbed) | 0-30s | 30-120s | > 120s |
| Gate poll_seconds (humanize) | 60-300s | 300-600s | > 600s |

A run that's TRUE_GREEN with 1 iter, 30s gate poll, $0.50 cost is the gold standard. We should see most testbed runs in that shape.

## 12. Operational notes

- **Cost guardrail**: the runner enforces a total $200 cost cap per 48h soak by reading `tasks.tokens_used` aggregates every iteration. SIGTERM if breached.
- **Slack/email noise**: the testbed runs have no Slack channel, so notifier warnings don't reach humans (verified in v1.7.2.4 logs). Humanize runs likewise.
- **Cleanup**: at soak end, close any open testbed PRs (the trigger scripts already do this on success). Manual sweep for any orphaned PRs from runner crashes.
- **Resumption**: if the runner dies mid-soak, `soak-state.json` lets it resume from the next slot. The metrics file appends, so no data is lost.
