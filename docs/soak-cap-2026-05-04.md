## Root cause: TL emits "fuzzy diffs" `git apply` rejects

The two `diff_apply_check_failed` runs (`4e7f2ca7`, `158f499c`) AND likely the `turn_cap_reached` one too (TL retried and ran out of turns) all hit the **same TL output shape**.

TL emits `apply_diff` steps with **placeholder hunk headers** — just `@@\n` instead of the required `@@ -L,N +L,N @@`. Example from `158f499c`:

```diff
--- a/tests/test_math_ops.py
+++ b/tests/test_math_ops.py
@@                                                              ← MISSING line numbers
-from calc.math_ops import add, divide, multiply, subtract
+from calc.math_ops import add, average, divide, multiply, percentage, subtract
@@                                                              ← MISSING line numbers
 def test_divide_by_zero_raises():
     with pytest.raises(ZeroDivisionError):
         divide(1, 0)
+
+
+def test_percentage():
+    assert percentage(1, 4) == 25.0
+    ...
```

`git apply` requires line numbers in hunk headers. Without them: `error: No valid patches in input (allow with "--allow-empty")`.

Identical shape in `4e7f2ca7` iter-2.

### Why this surfaced now

The TL prompt at `phalanx/agents/cifix_techlead.py:244` is too permissive:

```
apply_diff     — {"action":"apply_diff","diff":"<unified diff text>"}
```

`<unified diff text>` doesn't constrain hunk-header format. GPT-5.4 produces a human-readable "fuzzy" form (no line numbers) that's common in chat / docs but invalid for `git apply`.

In Phase 2 yesterday, TL happened to use `replace` steps for the simple cases. The coverage cell often needs `apply_diff` (multi-hunk add to a test file) and that's where the fuzzy-diff pattern emerges — repeatedly.

### Fix options (deferred)

- **(A) Strengthen the TL prompt** to require proper hunk headers:
  > apply_diff diffs MUST be valid input to `git apply`. Hunk headers MUST include line numbers in the form `@@ -L,N +L,N @@`. Empty `@@` markers are not accepted.
- **(B) Validate diff format in the plan validator** before dispatching the engineer step. Reject any apply_diff whose diff body has `@@\n` (no line numbers). Forces TL to re-plan.
- **(C) Engineer-side recovery**: try `git apply --recount` (rebuilds hunk counts from context lines) if the standard apply fails. Simpler patch; more lenient acceptance.

Most defensive: **(A) + (B) combined**. (A) prevents the issue at source; (B) catches drift if a future TL prompt tweak loosens it.

## Other findings

### flake cell exposes 3 distinct TL "wrong strategy" shapes

`ee5fd137` (regression), `df827eb2` (test deletion), `d53b74b9` (empty affected_files) — three different ways TL can mis-strategize on a flake. All three were correctly blocked by guardrails. None false-shipped. But the SHIP rate is 0/4 because TL keeps picking unsafe paths.

The right TL strategy for the flake cell is to **remove the artificial sleep**, not delete the test, not declare "I don't know," not break a different check. TL prompt may benefit from a flake-specific hint, but this is a separate intervention from the diff-format fix.

### coverage cell often has TWO failing checks

The `04-coverage-drop.patch` introduces unformatted code that breaks **both** Lint AND Test+Coverage. iter-1 typically fixes Lint; gate then catches `NOT_FIXED` because Test+Coverage is still red; iter-2 needs to add tests. iter-2 is where the diff format bug bites.

### Runner glitch (1/17 runs)

One flake run came back with `null` verdict / `null` run_id. The script likely emitted a non-standard summary line on that path; the runner's regex matcher dropped it. Low priority — fix the regex match in `_parse_result_line` to be more lenient OR have the runner SSH-query DB by recent timestamp if no result line found.

## What this means for marketplace launch

**The architectural readiness is unchanged.** Hard thresholds (false ships, sha mismatches, unsafe ships, commander exceptions, runtime cap) all hold. v1.7.2.4's gate is doing exactly what it was designed for.

**TL quality is the launch blocker.** Until TL reliably emits valid diffs (or the engineer/validator can recover from fuzzy ones), the SHIP rate on cells that need multi-line additions will be too low for customers.

### Pre-launch fix list

1. **TL diff-format prompt + plan-validator check** (Fix A + B above) — closes the diff_apply_check_failed shape entirely. Estimate: 1-2h.
2. **flake-strategy nudge** — small TL prompt addition: "for timing-sensitive tests / flake fixes, prefer fixing the source of timing variance (sleep, sampling), not deleting the test." Estimate: 30 min.
3. **Runner result-line parsing fix** — make `_parse_result_line` more tolerant. Estimate: 15 min.

After these fixes, re-run a 24h soak. Expect ≥80% SHIP rate as the new floor.
