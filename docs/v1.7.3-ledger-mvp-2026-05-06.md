## 1. What this MVP was for

The v1.7.3 vision is "shadow-mode validation on real OSS repos" — Phalanx
observes real CI failures, generates fixes, but does NOT push code.
Instead, it records what it would have done and stores evidence in a
ledger so we can compare against what the maintainer actually did
later.

The MVP scope explicitly stopped short of automation:

- shadow_ledger table + `runs.shadow_mode` flag
- engineer must NOT push in shadow mode
- store proposed_patch, verdict, confidence, root cause, affected files,
  cost, time
- CLI: `run`, `show`, `export`
- **No** poller, **no** ground-truth scraper, **no** cron, **no**
  matcher automation

This doc reports what the MVP architecture actually did when given 4
real GitHub workflow failures.

## 5. What this MVP definitively proves

### A. Shadow mode is push-safe end-to-end

Across two distinct engineer code paths — v1.6 Sonnet fallback AND
v1.7 step interpreter (the dominant production path) — and across
multiple verdict outcomes (SHIPPED_PROPOSED, SAFE_ESCALATE, FAILED),
shadow mode produced **zero git mutations** on every monitored repo.
The pre-run audit caught and closed 4 push paths before the first
real entry. The post-run audit confirmed nothing leaked.

### B. The calibration validator works on real-world data

Path B (anyio) is the load-bearing proof here. The v1.7.2.9 calibration
validator has been tested on synthetic reverts before. anyio is the
first time it fired on a bug we did not author, on a repo we don't
control, with clean architectural semantics:

- TL produced an accurate root_cause and pinned the affected file
- TL hedged at 0.4 (the exact problem v1.7.2.9 was built to catch)
- Validator rejected → run terminated → nothing shipped
- Engineer never ran → no possibility of unsafe code

### C. find_symbol enforcement and tool-call discipline hold under load

Three of four runs stayed at or below 9 tool calls (well under the 15
cap). One (humanize Path 2) emitted at 9 calls with a complete
diagnosis at confidence 0.95 in 76 seconds. The TL efficiency work
from v1.7.2.8/v1.7.2.9 transferred cleanly to shadow runs.

### D. Ledger captures TL evidence even when downstream fails

Entry 2 (Challenger hang) is the strongest case: TL emitted a precise
fix_spec at 0.95 confidence; the engineer never ran due to an
unrelated Celery infra issue; the ledger STILL has the diagnosis,
affected_files, and tool-call count. The verdict is FAILED but the
evidence is preserved. This means even imperfect runs feed the
ground-truth comparison workstream usefully.

## 6. What this MVP does NOT yet prove

Brutal honesty for a staff+ reader.

### Hit rate is undetermined at n=4

One SHIPPED_PROPOSED out of four runs is not a hit rate — it's an
anecdote. The v1.7.3 vision target was "≥70% of dispatched runs land
SHIPPED + SAFE_ESCALATE." We're at 75% (3/4) for that union, but the
sample is too small to be statistically meaningful. To call this
"hit rate evidence" we need ≥30 runs across ≥5 distinct repos.

### Stale historical workflow runs degrade TL accuracy

Entry 1 (humanize Path 1) on a 2-day-old workflow run: TL spent 8
tool calls without nailing the file body — historical PR/main-branch
context drifted. Path 1 had previously shipped clean on v1.7.2.7
when run fresh. Implication: shadow mode evidence should be collected
on FRESH failures (within hours), not stale ones. Future scoring
should weight by workflow-run age.

### Celery hang is a real systemic issue

Entry 2 (humanize Path 2) hit it once in TL. Entry 2's retry hit it
again in Challenger. Two distinct hangs in one MVP exercise = real
intermittent infra defect, not a one-off. **This will bite at scale.**
Fixing it requires Celery instrumentation + automatic abort logic and
is properly its own workstream (out of scope for the ledger MVP).

### One third-party run is one third-party run

Path B (anyio) closed the credibility gap to "we can do this safely
on a repo we don't own." It does not close the gap to "we have a
defensible hit rate against arbitrary maintainer-side bugs." That
needs Path B at scale — at least 5 distinct third-party repos × 5
distinct failure classes × ≥3 runs each.

## 9. Recommended next steps

In priority order, with explicit rejections for non-priorities:

1. **Fix the Celery hang.** Two hangs in 4 runs is a 50% repro rate at
   this scale. Will degrade the ledger to noise above ~10 entries.
   Celery worker hang detection + auto-abort. This is a real
   workstream, ~1-2 days.

2. **Run 5–10 more Path B entries on different third-party repos** —
   `Textualize/rich`, `python-attrs/attrs`, `pydantic/pydantic` (Rust
   core notwithstanding), `tiangolo/typer`, `pylint-dev/pylint`. Same
   cost envelope. The goal is converting "anecdote" to "signal" on
   third-party hit rate.

3. **Add age-weighting to the matcher** when we build it. Stale
   workflow runs (>48h old) should be flagged in the ledger so they
   don't pull down the aggregate hit rate. A workflow_run_id created_at
   timestamp is already in the GHA API.

### Explicitly NOT priorities

- Auto-poller / scheduler. Premature — manual dispatch covers ledger
  collection at this volume. Add poller only after 30+ entries prove
  the architecture stable.
- Ground-truth scraper. Premature — manual comparison fits the first
  ~10 entries. Build the scraper once we have hit-rate signal.
- UI / dashboards. Out of scope per the spec. The ledger IS the
  dashboard.
- New languages. Python-only proof first.
