## 1. What was hardened

The v1.7.3-ledger MVP architecture was correct — `1 SHIPPED_PROPOSED`,
`2 SAFE_ESCALATE`, `0 false ships` across 4 real runs. But two
runtime defects appeared:

1. Two distinct Celery worker hangs (humanize Path 2 first run + retry)
2. The reaper at 30-min TTL was too coarse; runs sat IN_PROGRESS for
   minutes before detection.

This release adds a heartbeat-aware stuck-task detector, commander
watchdog, guaranteed sandbox cleanup, structured observability, and
infra-specific failure classifications — without adding new agents,
UI, or changing CI-fixer reasoning behavior.

## 6. Architecture changes summary

### Before v1.7.3 hardening

```
Task PENDING → IN_PROGRESS (set by advance_run)
             → COMPLETED / FAILED (set by agent)
             OR
             → stays IN_PROGRESS forever if Celery hangs
                                          ↓
                       30-min reaper eventually marks FAILED
                                          ↓
                       Run.failure_class = NULL (no discriminator)
                                          ↓
                       Sandbox container leaks until manual cleanup
```

### After v1.7.3 hardening

```
Task PENDING → IN_PROGRESS + last_heartbeat_at + ttl_seconds (advance_run)
             → record_heartbeat() per LLM turn / per step
             → COMPLETED / FAILED (set by agent)
             OR
             → heartbeat goes stale > ttl_seconds
                                          ↓
                       2-min stuck-task detector sweep
                                          ↓
                       Task TIMED_OUT
                                          ↓
                       Run.status=FAILED, failure_class=
                       FAILED_INFRA_WORKER_HANG, error_message preserved
                                          ↓
                       Sibling PENDING/IN_PROGRESS tasks → CANCELLED
                                          ↓
                       Sandbox container stop_sandbox()
                                          ↓
                       runtime.run_finalized + runtime.sandbox_cleanup events
```

Plus a separate guard at the commander level: `_MAX_WAIT_SECONDS`
(45 min) caps the commander's poll-for-terminal loop. Past that,
`failure_class=FAILED_INFRA_TIMEOUT` is written and downstream tasks
are cancelled.

## 7. Detection latency budget

For any single hung task, worst-case detection is:

```
ttl_seconds (per role) + sweep_interval (120s) + transaction round-trips
```

| Role | TTL | Worst-case detection |
|---|---|---|
| cifix_techlead | 90 s | ~3.5 min |
| cifix_engineer | 180 s | ~5 min |
| cifix_sre / sre_setup / sre_verify | 300 s | ~7 min |
| cifix_challenger | 60 s | ~3 min |
| cifix_commander | 120 s | ~4 min |

For invite-only beta scale (low daily run count), this is well under
the previous 30-min reaper baseline.

## 8. Remaining risks (honest)

### R1. Celery worker process keeps running after detector marks task TIMED_OUT
The stuck-task detector writes the terminal state immediately, but
the actual Celery worker process may still be wasting compute until
its `SoftTimeLimitExceeded` fires (e.g., 10 min for TL, 15 min for
engineer). Acceptable for invite-only beta — the orchestration is
unblocked; only wasted CPU cycles remain. To address fully, would
need Celery worker introspection + revoke logic. **Not a beta blocker.**

### R2. Initial-heartbeat-window false positives
A worker that crashes BEFORE its first `record_heartbeat()` call has
its heartbeat stamped at dispatch time (in `advance_run`). The
detector still catches it within TTL + sweep, but the error message
("heartbeat stale 90s+") is misleading — the worker may have died
immediately after dispatch with no progress at all. Cosmetic.
**Not a beta blocker.**

### R3. self_critique_inconsistent should reclassify as SAFE_ESCALATE
S4 demonstrated that the TL self_critique gate (older v1.6.0 mechanism)
rejecting an emit at 0.76 confidence currently lands the run as
plain FAILED. Like the calibration_failed reclassification we did
earlier, this is the architecture refusing-to-ship — semantically
SAFE_ESCALATE. Verdict-classifier refinement, not a hardening defect.
**Tracked as separate workstream.**

### R4. Synthetic test rows leak into prod DB
The proof script INSERTs synthetic Project + Run + Task rows with
slugs starting with `cifix_runtime_proof_<ts>`. These accumulate
across proof runs. Cleanup pass needed:
```sql
DELETE FROM tasks WHERE run_id IN (SELECT id FROM runs WHERE work_order_id IN (SELECT id FROM work_orders WHERE description='synthetic'));
DELETE FROM runs WHERE work_order_id IN (SELECT id FROM work_orders WHERE description='synthetic');
DELETE FROM work_orders WHERE description='synthetic';
DELETE FROM projects WHERE slug LIKE 'cifix_runtime_proof_%';
```
**Bookkeeping; trivial.**

### R5. No automated stuck-detector dashboard
The detector writes `runtime.task_timeout` events, but there's no UI
or alert routing yet. Operator visibility is via log search or DB
query. **Out of scope per spec; ledger.json export covers the audit
need.**
